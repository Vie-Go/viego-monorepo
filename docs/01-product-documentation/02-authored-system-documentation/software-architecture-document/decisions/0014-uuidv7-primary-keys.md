# ADR 0014 — Uniform UUIDv7 primary keys

- **Status:** Accepted · **Date:** 2026-07-24 · **Deciders:** VieGo team

## Context

[Spec 003](../../../../../specs/003-modular-database-schemas/spec.md) split the database into five
module-owned schemas ([ADR-0002](0002-modular-monolith-with-spring-modulith.md)) and, at the same
time, adopted a **hybrid primary-key strategy**:

- `identity.explorers` / `auth_providers` → 16-byte `UUID`, for unguessability on public ids;
- `content` / `engagement` / `social` and `exploration.places|collections` → 8-byte **TSID**
  (`BIGINT`), for index size and time-ordered inserts;
- `exploration.provinces` / `wards` → natural ISO administrative codes.

Reviewing it before the first real feature (v0.2 Identity) surfaced four problems.

**1. The saving it optimises for is negligible at our scale.** `social.feed_entries` is the only
table where key width matters — fan-out at ~5k DAU × ~50 friends is roughly 90M rows/year. Row
widths, including the 24-byte tuple header:

| Strategy | `feed_entries` row | Heap @ 90M rows | `idx_feed_entries_timeline` entry |
|----------|--------------------|-----------------|-----------------------------------|
| All TSID | ~73 B | 6.6 GB | 32 B |
| Hybrid | ~90 B | 8.0 GB | 40 B |
| All UUIDv7 | ~105 B | 9.5 GB | 40 B |

The full spread is ~3 GB/year on the largest table — around a dollar a month of managed Postgres.

**2. The hybrid did not improve the index that matters.** `/feed/me` is served by
`idx_feed_entries_timeline(subscriber_id, feed_type, published_at)`, which leads with a
**16-byte explorer UUID** under the hybrid *and* under uniform UUIDv7 — identical. The hybrid paid
a complexity cost for a narrower PK index that is not the hot path. The same applies to
`idx_beats_explorer_captured` and `idx_notifications_recipient_unread`.

**3. The stated security rationale was inconsistently applied.** UUIDs were chosen for Explorers
"to prevent enumeration", but `/beats/{beatId}` and `/beats/{beatId}/reactions` expose **TSIDs**,
which are 42 bits of timestamp + node + a sequential counter — adjacent ids are guessable. A TSID
also leaks creation time, exactly like UUIDv7, while additionally being enumerable.

**4. Two key types is a permanent tax.** Two base entity classes, mixed reasoning at every join,
and — because a 64-bit id exceeds `Number.MAX_SAFE_INTEGER` — every TSID must be serialized as a
string or the React Native client silently rounds it. That failure mode does not throw; it opens
the wrong Beat.

Additionally, `TSID.fast()` derives a 10-bit node id per process; on a managed container platform
that requires explicit per-instance node configuration to stay collision-free.

## Decision

**Every generated primary key in the system is a UUIDv7** (RFC 9562): a 48-bit millisecond
timestamp followed by 74 random bits, **generated by the application**, never by the database.

- Implemented once in `com.viego.shared.domain.BaseEntity` via Hibernate's
  `@UuidGenerator(style = VERSION_7)` (available in Hibernate 7.1, which Spring Boot 4.0 resolves).
- Every logical cross-schema reference (`explorer_id`, `beat_id`, `place_id`, …) is `UUID`.
- **Natural keys are kept where they exist**: `exploration.provinces` and `exploration.wards`
  remain on ISO administrative codes (`VN-HN`). Reference data with a stable external identifier
  does not get a surrogate key.
- Entities whose identity *is* another entity's key — `identity.preferences`,
  `engagement.streaks` — keep `explorer_id` as their primary key rather than gaining a surrogate.
- The `io.hypersistence:hypersistence-tsid` dependency is removed.

This supersedes the hybrid strategy in
[spec 003](../../../../../specs/003-modular-database-schemas/research.md).

## Rationale

The decision rule is **one key type, time-ordered, chosen for operational safety rather than
bytes** — because at VieGo's scale the bytes do not decide anything.

- **Time-ordered writes.** The 48-bit timestamp prefix keeps B-tree inserts at the right edge of
  the index: no page splits, no fragmentation, good cache locality. This is the property TSID was
  originally chosen for, and UUIDv7 has it too.
- **Unguessable.** 74 random bits after the timestamp. Enumeration and IDOR probing are not
  possible, on *every* entity — not just Explorers.
- **Zero coordination.** No node ids, no sequence contention, no DB round-trip. Ids are assigned
  in the application, which also means an aggregate has its identity before it is persisted, and a
  module can be extracted to its own service without touching key generation.
- **No client-side precision hazard.** UUIDs are strings in JSON. There is no 53-bit cliff.
- **One type end to end.** One base class, one mental model, no "which kind of id is this?".

The only information UUIDv7 discloses that UUIDv4 does not is **row creation time**. For a travel
check-in app this is not sensitive, and public-facing references use the Explorer's `@handle`
(`/friends/add/{handle}`), not the UUID.

## Alternatives considered

- **Keep the TSID/UUID hybrid.** Rejected: buys ~1.4 GB/year on one table, does not improve the
  hot composite indexes, leaves two key types and the JSON precision hazard in place permanently,
  and applies its own anti-enumeration rationale inconsistently.
- **TSID everywhere.** The tightest schema (~1.4 GB/year better than the chosen option), and
  rejected for that reason being too small to matter against three real costs: enumerable ids on
  security-critical tables, mandatory per-pod node configuration, and the 64-bit JSON hazard
  multiplied across every id in the system. Revisit only if measurements show we are index- or
  storage-bound.
- **UUIDv4 everywhere.** Rejected: random ids scatter inserts across the whole index, causing page
  splits and bloat — the problem this schema explicitly set out to avoid. UUIDv7 is strictly
  better here with no offsetting cost.
- **`BIGSERIAL` everywhere.** Rejected: sequence contention, trivially enumerable, requires a DB
  round-trip before an aggregate has identity, and does not survive extraction into a service.

## Consequences

**Positive**
- One key type across five schemas; `BaseEntity` is the single place key generation is defined.
- Ids are unguessable everywhere, so ID opacity is no longer a per-table judgement call.
- No `tsid.node` configuration to get wrong when scaling out.
- Ids serialize as strings natively — no precision trap for the mobile client.

**Negative / accepted**
- ~1.4 GB/year more storage on `social.feed_entries` than an all-TSID schema, growing with fan-out.
- Row creation time is inferable from any id.
- 16-byte keys are wider in every index than 8-byte alternatives.

**Neutral**
- Ids are **not a security boundary.** Audience enforcement (`friends` vs `public`) is done in the
  service layer against the source of truth ([content design](../design/content.md)); unguessable
  ids are defence in depth, never the control.
- The real scaling lever for the feed is **fan-out-on-write vs. fan-out-on-read**, not key width.
  If `feed_entries` growth becomes the constraint, revisit the projection strategy first.
- Postgres 18 ships a native `uuidv7()`; we are on Postgres 16 and generate in the application, so
  no migration is implied either way.

## Follow-ups

- `exploration.places` now has a generated UUIDv7 key, but
  [data-schemas](../../../01-core-specifications/api-system-specifications/data-schemas.md#place-poi)
  describes Place `id` as a stable slug (`giang`, `halong`) and province ids as GeoJSON numbers.
  The ingestion mapping between the canonical datasets and these keys is **still undefined** and
  must be settled by the P2 Exploration dataset spec.
